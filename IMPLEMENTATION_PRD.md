# 产品导师风格项目诊断工具 - 落地版 PRD

## 1. 执行摘要 (Executive Summary)

### 1.1 产品目标
为 Vibe Coding 用户和黑客松开发者提供一个“情绪化”的项目诊断工具。通过模拟顶级产品导师的毒舌风格，对用户的项目文档和代码结构（Vibe）进行 Critique，输出具有逻辑深度的方向性建议，主打“情绪价值”。

### 1.2 核心痛点
- 开发者自嗨：写了一堆代码和文档，但不确定逻辑是否自洽。
- 缺少反馈：没人泼冷水，不知道哪些功能是冗余的。
- 情绪需求：开发者需要一种“被大佬点评”的仪式感。

---

## 2. 交互细节 (Interaction Details)

### 2.1 输入阶段 (Input Flow)
- **文件上传**: 强制要求上传 `README.md` 或 `Manual.md`。
- **Vibe 输入 (可选)**: 提供一个文本框，用户可粘贴通过 `tree` 命令生成的项目目录树，或粘贴 `package.json` 的依赖列表。
- **导师选择**: 8 位导师的头像/卡片展示，点击选择（默认选中 Karpathy）。
- **状态切换**: 用户需选择项目阶段：`Idea/Prototype` 或 `Launched/Existing`。

### 2.2 审判阶段 (Analyzing State)
- **视觉反馈**: 全屏毛玻璃遮罩，展示“导师正在审阅”的动效。
- **文案轮播**: 根据所选导师切换文案（如马斯克：“正在检查你的第一性原理...”；乔布斯：“正在砍掉你那些愚蠢的按钮...”）。

### 2.3 结果展示 (Result Flow)
- **灵魂初印象 (Vibe Check)**: 置顶展示导师的一句话评价。
- **逻辑推导区 (Logical Flaw)**: 采用“折叠面板”或“高亮标注”形式，指出文档中的逻辑矛盾点。
- **6 维 Vibe 图**: 采用雷达图或条形图展示评分，维度根据项目阶段动态调整。
- **方向建议 (Directions)**: 3 条加粗的宏观建议，**不提供代码 Prompt**。

---

## 3. 状态机定义 (State Machine)

| 状态 | 触发条件 | UI 表现 | 备注 |
| --- | --- | --- | --- |
| **IDLE** | 页面加载 | 显示上传区域、导师列表、阶段选择 | 初始状态 |
| **PARSING** | 文件上传后 | 上传进度条 | 提取文本内容 |
| **ANALYZING** | 点击“开始审判” | 导师专属加载动画 + 风格化轮播文案 | 触发后端 AI 逻辑 |
| **SUCCESS** | AI 返回数据 | 渲染结果页，展示 Critique 报告 | - |
| **FAILED** | 接口报错/超时 | 展示导师风格的报错文案（如：“你的项目烂到我的 AI 都罢工了”） | 提供重试按钮 |

## 4. 数据规范与 AI 策略 (Data Spec & AI Strategy)

### 4.1 核心数据结构
```json
{
  "projectId": "uuid",
  "mentor": "musk | jobs | karpathy | ...",
  "stage": "prototype | launched",
  "inputData": {
    "docText": "string",
    "fileTree": "string (optional)",
    "techStack": ["string"]
  },
  "critique": {
    "vibeCheck": "string",
    "logicalFlaw": {
      "point": "string",
      "reasoning": "string"
    },
    "dimensionScores": {
      "interaction": 0-100,
      "scope": 0-100,
      "vibe": 0-100,
      "value": 0-100,
      "hook": 0-100,
      "share": 0-100
    },
    "directionalAdvice": ["string"]
  }
}
```

### 4.2 AI 诊断策略 (Prompting Strategy)
- **逻辑漏洞检测**: 要求 LLM 提取文档中的“产品承诺”与“功能实现”，寻找不匹配项（如：承诺极简但配置极其复杂）。
- **Vibe 评估逻辑**:
  - 若有 `tree` 结构：分析目录深度、命名规范、是否有测试目录。
  - 若有 `techStack`: 分析工具选型是否过度封装（Over-engineered）。
- **情绪控制**: 导师 Persona 必须保持 80% 的攻击性和 20% 的洞见感。

---

## 5. 成功指标 (Success Metrics)

1. **分享卡片生成率**: 用户生成并点击“下载/保存”卡片的比例。
2. **多导师尝试率**: 同一项目被不同导师“审判”的平均次数（衡量娱乐性）。
3. **完成度分析触发率**: 用户主动粘贴 `tree` 结构的比例。

## 7. 风险与边界 (Risks & Boundaries)

- **隐私问题**: 明确告知用户不建议上传包含敏感 API Key 的代码文件，只建议上传 README 或 Tree。
- **AI 幻觉**: 导师可能会指出一些不存在的逻辑矛盾，需在界面下方标注“以上评价仅供娱乐与启发”。
- **文件限制**: 最大支持 5MB 的 Markdown/PDF，超出后前端拦截。
